Sprint XL livré le 2026-05-05 · 4 vagues HTTP + P2P

✅ Audit 80/100 21/21 tests 14 fichiers C++ 15 fichiers JS/HTML/CSS 4 nouveaux tests unit

Contexte

Suite à l'audit utilisateur post-V1.6.4, 5 zones d'instabilité ont été identifiées dans les 2 chemins de transfert web (HTTP visiteur↔host et P2P visiteur↔visiteur via WebRTC) :

  1. Aucune stratégie de resume (pas de Range Request, OPFS perdu au reload, pas d'iceRestart)
  2. Téléchargements bloqués à 0% (event progress émis trop tard)
  3. Sessions web trop strictes (TTL 30s, cookie session pur, redémarrage host = perte totale)
  4. PIN jamais mémorisé côté browser
  5. Aucun historique persistant côté host

Wave 1 — Stabilité HTTP

A · Bug « bloqué à 0% »

TransferProgressEvent est désormais émis dès le début du handler streamFile/streamZip, avant set_content_provider. Avant le fix, l'event n'arrivait qu'au 1er chunk lu — une coupure SSL handshake ou un browser bloqué sur le warning cert HTTPS laissait l'UI desktop figée à 0% sans signe de vie.

B · Cancel flag leak

WebService::releaseCancelFlag(sessionId) nettoie la map cancelFlags_ à toutes les terminaisons du content_provider (cancel utilisateur, EOF normal, sink.write fail).

C · Resume upload (version minimale)

upload.js retry automatiquement 3 fois avec backoff exponentiel 1s/2s/4s. Si après 3 retries l'upload échoue, un bouton « ↻ Reprendre » manuel apparaît dans l'UI.

Limitation V1.6.5 : le serveur ne supporte pas encore le vrai resume avec offset HTTP Content-Range. Chaque retry repart from scratch — efficace pour les coupures < 7s cumulées. Reporté à V1.6.6.

D · HTTP Range Requests (RFC 7233)

download_routes.cpp parse le header Range: bytes=N-, seek dans le fichier, retourne 206 Partial Content avec Content-Range. Le browser fait automatiquement le retry avec range en cas de coupure réseau pendant un download <a href download>.

Module range_parser.{hpp,cpp} avec test unit (11 cas).

Wave 2 — Resume P2P

G · iceRestart automatique

p2p_transport.js monitore pc.connectionState. Si disconnected > 5s (avant le cleanup à 15s), le sender appelle pc.createOffer({iceRestart:true}) et envoie un signal ice-restart au pair. Le receiver applique setRemoteDescription + createAnswer + signal ice-restart-answer. Symétrie complète.

Si reconnexion < 15s : le DataChannel reprend depuis l'offset courant. Sinon : cleanup normal « ✗ Wi-Fi perdu ».

E · Sidecar IndexedDB receveur

Nouveau module idb.js (~80 lignes, API native sans dépendance) expose window.LTR.idb. À chaque ack reçu (toutes les 500ms), p2p_session.js persiste dans le store ltr-p2p-pending : 

{
  opfsName, peer, fileMeta,
  bytesWritten, lastAckAt, startedAt
}

Au boot d'un nouvel onglet, transfer_registry.js scan le store : pour chaque entry > 30s ancienne (donc orpheline), affiche une card « Interrompu (X/Y MB) » avec deux boutons :

  • ▶ Reprendre : récupère le partial OPFS via getFileHandle.getFile() et déclenche un download Blob
  • ✕ Annuler : purge OPFS + sidecar IndexedDB
Limitation V1.6.5 : le « Reprendre » récupère le partial téléchargé, pas une vraie reprise du DataChannel (qui demanderait coopération du sender pour renégocier la session avec offset). Reporté à V1.6.7.

Wave 3 — Sessions résilientes

H · Cookie persistent HMAC ltr_remember

TTL session inactivité : kWebSessionTtl 30s → 300s (5 min). Nouveau cookie persistent ltr_remember Max-Age 30 jours, format : 

{deviceId}.{exp}.{pinHash8}.{HMAC_SHA256_HEX}

HMAC SHA-256 RFC 2104 implémenté manuellement au-dessus de picosha2 (pas de nouvelle dépendance, conforme CLAUDE.md). Secret HMAC = fingerprint cert HTTPS V1.6.4 (déjà persisté dans cfgDir/cert.pem, stable). Compare HMAC constant-time dans verifyPersistentToken pour éviter les timing-attacks.

Émis par POST /api/auth quand body contient remember:true (checkbox cochée par défaut). Endpoint NEW POST /api/auth/refresh vérifie le cookie + recrée la session sans re-PIN. Logout efface 4 entrées : ltr_token, ltr_remember, ltr-pin-encrypted, ltr-pin-iv.

Test unit test_persistent_token (9 cas : roundtrip, PIN différent, secret différent, expiré, tampered, format invalide).

I · « Mémoriser le PIN » WebCrypto AES-GCM

Module pin_storage.js chiffre le PIN avec AES-GCM 256, clé dérivée via HKDF SHA-256 du fingerprint cert HTTPS. Nécessite un secure context (HTTPS ou localhost) — la checkbox est désactivée en HTTP plain.

À l'ouverture suivante de /login : si entry présente + fingerprint identique, le PIN est déchiffré et pré-remplit silencieusement les 6 cases. L'utilisateur clique simplement « Se connecter ».

Placeholder V1 : la clé HKDF dérive d'un secret PUBLIC (le fingerprint cert est exposé via /api/cert-info). Ne protège QUE contre la lecture passive du fichier localStorage par un autre user du même device. Pas robuste contre XSS / vol device / malware. À durcir avec WebAuthn ou une CryptoKey non-extractable pour V2.

Wave 4 — Historique persistant

J · peers_history.json côté host

Nouveau module infra/peers_history.{hpp,cpp} sur le pattern KnownPeers V1.6.4. Stockage : ~/Library/Application Support/LocalTransfer/peers_history.json (Mac), %APPDATA%/LocalTransfer/peers_history.json (Windows).

Schéma entry :

{
  deviceId, name, platform,
  kind: "native" | "web",
  fingerprint,
  firstSeen, lastSeen,         // epoch seconds
  totalTransfers, totalBytes
}

Hooks AppController::onEvent : PeerSeenEvent → touch(), TransferDoneEvent → recordTransfer(bytes). Purge auto > 30 jours au load (décision BA). Méthode forget(deviceId) pour clic-droit « Oublier ».

Test unit test_peers_history (7 cas).

K · transfer_history.json côté host

Nouveau module infra/transfer_history.{hpp,cpp}. Stockage : cfgDir/transfer_history.json.

Schéma entry :

{
  sessionId, peerDeviceId, peerName,
  kind: "tcp-out" | "tcp-in" | "http-up" | "http-down",
  fileCount, totalBytes,
  status: "pending" | "ok" | "failed" | "cancelled",
  startedAt, finishedAt, error
}

Cap 1000 entries (drop des plus anciennes si dépassement). Auto-purge entries > 6 mois au load.

Test unit test_transfer_history (8 cas dont cap 1000 vérifié à 1500 insertions).

Note : les transferts P2P browser↔browser ne sont pas loggés ici — le host ne voit que les signaux, pas les données. Seul le browser garde leur trace dans transfer_registry.js (localStorage).

L · TOFU P2P côté browser

À chaque offer P2P reçue, p2p.js::handleIncomingOffer extrait l'empreinte DTLS de la SDP (regex a=fingerprint:sha-256 ...) et lookup dans IndexedDB store ltr-p2p-known-peers :

  • 1re fois → set silencieux (TOFU)
  • Match → silencieux
  • Mismatch → toast bloquant showTofuToast avec deux actions : « Faire confiance » (set + accept) ou « Refuser » (signal refuse + cleanup)

Pas d'auto-dismiss : décision explicite obligatoire.
