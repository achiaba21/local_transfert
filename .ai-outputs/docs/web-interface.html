1. Vue d'ensemble

L'interface web embarquée ajoute à LocalTransfer un serveur HTTP interne (port 45456). Depuis le même réseau local, n'importe quel appareil (mobile, tablette, laptop, quel que soit l'OS) peut ouvrir une URL dans son navigateur et échanger des fichiers bidirectionnellement avec le host —  sans installation.

Les sessions navigateur authentifiées deviennent des « pairs » dans l'app desktop, au même niveau que les apps natives, dans une liste unifiée. L'utilisateur voit tout le monde au même endroit et peut envoyer indifféremment vers un pair natif ou vers un navigateur.

💡 En une phrase :
« Le host imprime un QR code + PIN dans sa fenêtre LocalTransfer. Les invités scannent, saisissent le PIN une fois, puis échangent des fichiers dans leur navigateur. »

2. Problème résolu & valeur apportée

Avant

  • Les pare-feux OS (macOS, Windows Defender) bloquent souvent les connexions TCP entrantes des apps non signées → découverte non symétrique entre 2 apps natives (A voit B, B ne voit pas A).
  • Un utilisateur qui veut envoyer un seul fichier doit quand même installer toute l'app.
  • Les appareils iOS / Android n'avaient pas d'accès à LocalTransfer.

Après

  • 1 seule permission pare-feu à accorder (côté host, pour le port 45456) — côté visiteur, le navigateur est déjà autorisé.
  • Depuis iOS, Android, ou n'importe quel navigateur : zéro install pour échanger.
  • Auto-propagation : si le visiteur est sur le même OS que le host, il peut télécharger l'app native directement (Windows .exe, macOS .app, Linux ELF).

3. Architecture haute-niveau

La couche ltr::web est sœur de ltr::network — elle coexiste sans toucher au protocole TCP natif (LTR1) existant. 

┌──────────────────────────────────────────────────────────────┐
│  UI (SFML) — ltr::ui                                         │
│    MainScreen : Sidebar | Centre | SharePanel (NEW)          │
│    DeviceListItem avec pill "Web" pour kind=PeerKind::Web    │
├──────────────────────────────────────────────────────────────┤
│  App — ltr::app                                              │
│    AppController::requestSend() dispatche selon peer.kind :  │
│       Native → TransferClient  (TCP LTR1, existant)          │
│       Web    → WebService::pushFiles (tickets + SSE)         │
├──────────────────┬────────────────────┬──────────────────────┤
│  Network         │  Web (nouveau)     │  Infra               │
│  (inchangé)      │                    │                      │
│  DiscoveryService│  HttpServer        │  Config              │
│  TransferServer  │  WebService        │  FilesystemService   │
│  TransferClient  │  WebSessionStore   │  HashService         │
│  Protocol LTR1   │  DownloadTicketStore                      │
│                  │  SseBroadcaster    │                      │
│                  │  SelfBinary (OS)   │                      │
│                  │  QrCode            │                      │
├──────────────────┴────────────────────┴──────────────────────┤
│  Domain — Device{kind=Native|Web, sessionToken}              │
│  Core   — EventBus (std::variant) — flux événements unifié  │
└──────────────────────────────────────────────────────────────┘

Invariant architectural : aucun nouveau type d'événement dans core::EventBus. Les sessions web réutilisent intégralement les événements existants (PeerSeenEvent, IncomingOfferEvent, TransferProgressEvent, etc.). L'UI desktop traite les pairs web sans modification logique — seule l'icône pill « Web » est ajoutée pour différencier visuellement.

4. Ports réseau

45454 UDP Discovery beacon natif (broadcast 255.255.255.255) — inchangé
45455 TCP Protocole LTR1 natif (offer/accept/chunks) — inchangé
45456 HTTP NOUVEAU — serveur web. Fallback automatique sur 45457–45466 si occupé.

5. Endpoints HTTP exposés

Méthode Route Auth Description
GET / non Page d'accueil (index.html embarqué dans le binaire)
GET /app.js, /style.css, /icons/*.svg non Assets statiques embarqués
GET /api/host-info non Nom du host, plateforme, URL de téléchargement binaire
POST /api/auth PIN Vérifie le PIN 6 chiffres, ouvre une session, retourne un cookie ltr_token
GET /api/me cookie Whoami — infos sur la session courante
POST /api/upload cookie Upload multipart vers le host. Émet les events UI (progress, done, failed)
GET /api/events cookie Server-Sent Events long-lived (host→browser)
GET /api/download/:ticketId cookie Stream d'un fichier via ticket one-shot (host→browser)
GET /download/self non Binaire natif de l'app courante (voir §7 Auto-propagation)

7. Auto-propagation du binaire natif

Chaque host sert sa propre version du binaire natif. Pas de bundle cross-OS — logique virale par-OS : un Mac propage un Mac, un Windows propage un Windows. Pour les cas cross-OS, la page web propose un lien vers GitHub Releases.

OS du host Contenu servi Mécanisme Content-Type
Windows LocalTransfer-Windows.exe GetModuleFileNameW sur le process courant application/vnd.microsoft.portable-executable
Linux LocalTransfer-Linux (binaire ELF) readlink /proc/self/exe application/octet-stream
macOS LocalTransfer-macOS.zip Zip du .app bundle via miniz (recursive) application/zip
⚠️ Sécurité — prérequis V2 : l'auto-propagation n'est sûre qu'avec signature des binaires (Apple Developer ID pour macOS, Authenticode pour Windows). Sans signature, un attaquant sur le LAN pourrait distribuer une version trojanisée. À régler avant release publique — non bloquant en V1 (usage contrôlé).

9. Sécurité & limites V1

Modèle de sécurité

  • PIN d'accès web : 6 chiffres aléatoires générés au démarrage de l'app (stable pour toute la session d'app). Distinct du PIN per-transfer 4 chiffres du protocole natif. Affiché dans la SharePanel desktop.
  • Cookie de session : HttpOnly + SameSite=Lax. Valide tant que le host tourne et que la session a été touchée dans les 30 dernières secondes (kWebSessionTtl).
  • Tickets de download : one-shot + TTL 5 minutes (kDownloadTicketTtl). Le ticket est vérifié contre le token de session (pas d'usage croisé).
  • Upload path traversal : le nom de fichier est désinfecté (path::filename()), l'écriture utilise FilesystemService::uniqueTargetPath() qui gère les collisions sans écraser (foo.txt → foo (1).txt).

Limites V1 assumées

LimiteRaisonReporté en
Pas de HTTPS (HTTP plain) Certificat self-signed = warnings navigateur pénibles V2
Pas de mDNS / Bonjour QR code + URL manuelle suffisent pour V1 V2
Sessions perdues au redémarrage app 100% in-memory. Redémarrage = re-PIN sur les navigateurs V2
Pas de rate-limit sur le PIN Risque mineur sur LAN de confiance V2
Pas de signature des binaires distribués Hors scope code. Apple Developer ID + Authenticode à acquérir. Requis avant release publique
macOS .app zippé en RAM complète Acceptable < 50 Mo. Streaming zip à prévoir si taille augmente. V2

Hypothèse fondamentale

Hypothèse LAN de confiance : V1 part du principe que le réseau local est de confiance (WiFi domestique/bureau, pas de WiFi public ouvert). Le PIN + les cookies cookie-session sont suffisants dans ce contexte. Pour des scénarios plus exposés (WiFi café, réseau d'entreprise hostile), V2 ajoutera HTTPS self-signed + vérification SHA-256 côté page web.

11. Changelog

V1.1.8 — 2026-04-24 LIVRÉE

Suppression du fichier zip temporaire pour les envois dossier host → web. Le zip est désormais assemblé à la volée dans la réponse HTTP, sans jamais toucher le disque. Premier octet reçu côté navigateur < 200 ms après le clic « Télécharger ».

  • ✅ ZIP streamé à la volée (StreamingZipSource) : format STORE + data descriptor, CRC32 calculé incrémentalement par chunk, Content-Length exact précalculé
  • ✅ Zéro fichier temp : /tmp/ltr-zip-*.zip disparaît complètement du flow, folder_zipper supprimé
  • ✅ Tickets rejouables (V1.1.8) : le visiteur peut retélécharger pendant les 15 min du TTL si rupture réseau ou mauvais clic (consume() → get() non-destructif)
  • ✅ Progress events throttlés : 100 ms / 1 MB pour ne pas flooder l'EventBus desktop
  • ✅ Test unitaire octet-exact : test_streaming_zip vérifie la formule Σ(92 + 2·nameLen + fileSize) + 22 + valide le zip avec miniz
  • ✅ 8/8 tests passent, 0 régression

V1.1.7 — 2026-04-23 LIVRÉE

Itération UX + performance : transferts de dossiers dans les deux sens, réactivité de l'app (thread background), modularité JS côté web.

  • ✅ Envoi dossier web → host (sans zip) : webkitdirectory → chemin relatif préservé → arborescence reconstruite (desktop + Android ; iOS masqué)
  • ✅ Zip host → web en thread background : app desktop 100% réactive pendant compression
  • ✅ Mode STORE (no compression miniz) : 5-10× plus rapide sur contenu binaire, taille quasi-identique
  • ✅ JS modulaire : common.js + upload.js + download.js + app.js (bootstrap)
  • ✅ Sanitize path traversal : rejet strict .., absolus, null bytes côté /api/upload
